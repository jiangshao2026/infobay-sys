## 2. 数据跨模块跨用户的持久化机制

### 2.1 设计原则

- **数据与 UI 分离**：业务数据通过 React Context + Hook 集中管理，页面组件仅消费和更新
- **基于浏览器 localStorage**：所有变更（新增/编辑/删除/审批）都写入浏览器本地存储
- **前缀隔离**：使用 `STORAGE_PREFIX`（当前为 `xb-demo-v4-`）隔离不同模块状态，版本升级时自动清除旧缓存
- **用户数据可见**：同一浏览器下，不同登录用户看到同一份数据（公共数据）

### 2.2 持久化实现

```typescript
// usePersistedState.ts - 通用 Hook
// 初始化时从 localStorage 读取；状态变化时写入
// 智能类型检测：数组/对象/基本类型的解析保护

export function usePersistedState<T>(key: string, initialData: T) {
  const [data, setData] = useState<T>(() => {
    try {
      const raw = localStorage.getItem(STORAGE_PREFIX + key)
      if (raw) return JSON.parse(raw) as T
    } catch {}
    return initialData
  })
  useEffect(() => {
    try { localStorage.setItem(STORAGE_PREFIX + key, JSON.stringify(data)) } catch {}
  }, [data, storageKey])
  return [data, setData] as const
}

// 重置函数：遍历所有前缀并清除

```

- **持久化**：当前登录用户信息持久化到 localStorage(key: xb-demo-current-user)

### 3.4 权限控制方式

系统采用**基于角色（Role-based）的权限控制**，核心逻辑为：

1. **角色字段**：通过 `currentUser.role` 字段
2. **审批按钮的条件渲染**：操作列按钮根据 `currentUser.role` 动态显示/隐藏
3. **审批链配置驱动**：不同业务类型定义了不同的审批链（`APPROVAL_CHAINS`）
4. **审批人角色**：在审批链配置中，每个节点定义了 `roles` 数组，只有对应角色的用户才能看到并操作该节点的"审批"按钮
5. **默认审批人默认为当前用户**：审批弹窗中 reviewer 字段默认填入当前用户姓名，且不可修改
6. **权限判断逻辑**：
   - 销售 → 仅销售可发起监理合同审批和建设合同审批
   - 监理工程师 → 仅监理工程师可审批监理规划、实施细则、质量问题等
   - 总监理工程师 → 同时具备监理工程师权限（可审批所有监理工程师审批节点
   - 部门经理 → 可审批合同管理中部门经理审批节点
   - 副总经理 → 可审批合同终审节点和知识文档

## 4. 分级审核流程（以监理合同审核和监理规划审核为例）

### 4.1 监理合同审核流程（4级审批）

#### 4.1.1 审批链

```
销售发起（销售） → 总监理工程师审批（总监理工程师） → 部门经理审批（部门经理） → 分管副总经理审批（副总经理）
```

#### 4.1.2 状态流转

| 节点 | 触发角色 | 操作名称 | 状态变化 |
|------|---------|----------|
| 1 | 销售 | 发起审批 | 待审批 → 待总监理工程师审批 |
| 2 | 总监理工程师 | 审批 | 待总监理工程师审批 → 待部门经理审批 |
| 3 | 部门经理 | 审批 | 待部门经理审批 → 待分管副总经理审批 |
| 4 | 副总经理 | 审批 | 待分管副总经理审批 → 已审批 |
| - | 任意节点 | 驳回 | 原状态 → 已驳回 |

#### 4.1.3 审批记录

- **审批人**：默认为当前用户姓名，不可修改
- **审批意见**：可填写自定义意见
- **审批状态**：通过 / 驳回
- **审批日期**：自动记录审批时间
- **审批链时间线**：合同详情页展示完整审批流程时间线

#### 4.1.4 销售发起弹窗（第一级特殊）

- 必须选择总监理工程师（必填）
- 销售意见（必填）
- AI 智能审核（15 秒模拟审核）
  - 随机返回 2-3 条风险提示（如首付款比例、违约条款对等性、监理范围描述、工期偏差、质量验收标准等）
  - 阅读风险后选择是否继续提交
- 提交成功后合同状态转为"待总监理工程师审批"

#### 4.1.5 标准审批弹窗（2-4 级）

- ReviewModal 组件
- 自动填充当前用户姓名作为审批人，禁用修改
- 审批通过后状态流转到下一个节点
- 驳回到"已驳回"

#### 4.1.6 按钮可见性规则

| 合同状态 | 可见角色 | 按钮文本 |
|----------|---------|--------|
| 待审批 | 销售 | 发起审批 |
| 待总监理工程师审批 | 总监理工程师 | 审批 |
| 待部门经理审批 | 部门经理 | 审批 |
| 待分管副总经理审批 | 副总经理 | 审批 |
| 其他状态 | 不限制 | 查看/修改/作废/收款（按状态显示） |

| **监理师管理** | 总监理工程师/监理工程师人员信息、监理证书信息 |
| **知识库** | 监理工程师/监理规划相关文档条目 |
| **系统管理** | 组织架构（部门/职位） |
| **系统管理** | 用户列表（系统用户，按姓名作为用户名） |
| **权限/角色配置 | 角色权限矩阵 |
| **数据重置** | 仅保留重置按钮+说明，其他已删除 |

### 5.4 演示数据生成逻辑

1. **主从一致性**：从数据（合同/监理规划/细则/质量问题/进度计划）必须关联到主数据项目编号（projectCode）
2. **逻辑合理性**：合同金额与项目投资额比例合理（监理费约为项目投资额的 1-3%）
3. **时间逻辑**：项目启动日期早于合同签订日期，合同结束日期晚于项目结束日期
4. **审批状态分布**：不同状态（待审批/审批中/已审批/已驳回/执行中/即将到期/已完成）分布均匀
5. **审批记录完整性**：已审批项目应包含完整的各节点审批人、意见和日期
6. **角色一致性**：审批人姓名必须从演示用户列表中合理选择

### 6.2 数据状态分布建议

- **项目数**：15-20 个建设项目
- **监理合同**：每个项目 1 份合同（或多个阶段合同
- **监理规划/细则**：每个项目 1-2 份
- **质量/进度/成本**：每个项目 3-8 条记录
- **变更/安全/信息**：每个项目 1-5 条
- **监理付款**：每份合同 2-4 条
- **验收/归档**：每个项目 1-3 条
- **组织/会议**：每个项目 2-5 条
- **总数据量**：约 200-300 条演示数据条目
