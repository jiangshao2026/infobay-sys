监理服务管理系统需求规格说明书
1.引言
1.1 目的
本文档旨在定义监理服务管理系统的功能与非功能需求，为系统设计、开发、测试和验收提供依据。系统依据 GB/T 19668 系列《信息技术服务 监理》标准，对信息化监理项目的全生命周期进行信息化、智能化管理，实现“四控三管一协调”的在线协同，并通过知识库与大模型能力赋能监理工作。
1.2 适用范围
- 用户群体：监理工程师、总监理工程师代表、总监理工程师、部门经理、分管常务副总经理。
- 终端类型：PC 端 Web 版，暂不考虑移动端。
- 核心业务：从监理合同登记、监理大纲/规划/细则编制，到过程控制（四控三管一协调）、项目验收、资料归档的全过程管理，以及知识库的构建与智能应用。
1.3 依据标准
- GB/T 19668.1-2014《信息技术服务 监理 第1部分：总则》
- GB/T 19668.2-2017《信息技术服务 监理 第2部分：基础设施工程监理规范》
- GB/T 19668.3-2017《信息技术服务 监理 第3部分：软件工程监理规范》
- GB/T 19668.4-2017《信息技术服务 监理 第4部分：信息安全监理规范》
- GB/T 19668.5-2018《信息技术服务 监理 第5部分：监理工作内容》
- 其他相关法律法规、标准规范（由系统知识库归集管理）
1.4 术语与缩写
- 四控：质量控制、进度控制、成本控制、变更控制
- 三管：合同管理、安全管理、信息管理
- 一协调：组织协调（沟通协调）
- RAG：检索增强生成（Retrieval-Augmented Generation）
- AI：人工智能，本系统特指基于大语言模型的智能辅助能力

2. 系统总体描述
2.1 系统目标
构建一个集流程化、规范化、智能化的监理服务管理平台，实现：
- 监理项目从合同登记到归档的全生命周期在线管理；
- 四控三管一协调的标准化流程与表单支撑；
- 基于知识库和大模型的合同智能评审、方案评审、监理意见辅助生成及智能问答；
- 监理知识的沉淀、共享与持续优化。
2.2 用户角色与权限模型
权限规则：
- 菜单级权限按角色分配，数据级权限按所属部门或项目范围控制。
- 操作级权限严格分层：提交、审核、签发、批准依次由低到高，禁止越权操作。
- 人员离职或岗位变动时，支持工作流委托与权限转移。
2.3 总体业务流程
监理项目生命周期划分为五个阶段：
1. 项目启动与合同登记：监理合同登记、审核生效、项目空间建立。
2. 监理准备阶段：编制监理规划、监理实施细则，组建项目监理机构。
3. 过程实施：执行四控三管一协调各项活动，记录与跟踪。
4. 验收与归档：受理验收申请、组织验收、出具报告、资料归档。
5. 知识库管理（贯穿全程）：知识采集、入库审核、维护更新、智能应用。

3. 功能需求
3.1 项目启动与合同登记
业务流程
1. 新建项目登记：部门经理指派人员新建项目，填写基本信息并上传监理合同电子版。
2. 智能识别与填报：系统自动识别合同关键字段（合同编号、委托方、承建方、监理范围、金额、工期等）并回填，人工补录确认。
3. AI合同智能审核（可选）：调用AI生成《合同智能审核意见书》，标示风险条款、缺失项。
4. 合同审核：部门经理审核合同内容与公司业务、资源匹配性，可采纳AI意见。
5. 项目生效：审核通过后生成项目编号，启动项目空间。
主要表单与操作
3.2 监理准备阶段
3.2.1 监理规划编制
1. 编制：总监理工程师使用模板创建规划，系统提供GB/T 19668.5要求的规划结构。可利用AI辅助生成初稿。
2. 审批：总监提交后，一般项目由部门经理审批，重大项目（合同额>100万或公司规定）需分管副总批准。
3. 生效：批准后规划基线化，锁定不可编辑。
3.2.2 监理实施细则编制
1. 编制：总监或代表创建细则，可关联规划章节，利用AI生成检查清单。
2. 审批：总监审核批准，必要时部门经理会签。
3. 发布：细则与后续质量控制、检查表单动态关联。
3.2.3 组建项目监理机构
1. 配置：总监从公司人员库选择角色成员，明确分工。
2. 报审：提交部门经理审核人员资质与工作量匹配性。
3. 发文：可生成《监理机构通知书》通知建设单位。
主要表单与操作
3.3 四控
3.3.1 质量控制
流程： 方案审核 → 过程检查 → 问题通知 → 整改闭环 → 测试见证 → 里程碑验收。
3.3.2 进度控制
流程： 计划审批基线化 → 定期进度跟踪 → 偏差预警 → 纠偏联系单 → 进度报告。
3.3.3 成本控制
流程： 承建方提交支付申请 → 监理审核工程量与质量 → 签发支付证书 → 资金台账动态更新。

3.3.4 变更控制
流程： 变更申请 → 影响分析（AI辅助估算） → 分级审批 → 签发变更指令 → 更新基线 → 实施跟踪闭环。
3.4 三管
3.4.1 合同管理
流程： 建立合同台账（含补充协议） → 履约节点自动提醒 → 索赔审查 → 违约记录。
3.4.2 安全管理
流程： 安全方案审核 → 安全检查记录 → 隐患通知 → 整改闭环 → 安全事件报告。
3.4.3 信息管理
流程： 文档分类存储 → 文件收发登记 → 版本管理 → 归档整理。
3.5 一协调：组织协调
流程： 会议议题自动汇总 → 会议纪要编制与签发 → 决议跟踪 → 工作联系单往来 → 协调看板预警。
3.6 验收与归档阶段
流程： 受理验收申请 → 审核是否具备条件 → 编制验收方案 → 验收测试记录 → 出具验收报告 → 监理工作总结 → 资料完整性检查与归档。
3.7 知识库管理与应用
3.7.1 知识内容分类体系
采用多级分类树+标签组织，一级分类包括：政策法规、标准规范、项目管理办法、方案审核意见模板、验收审核意见模板、监理工作案例、法律法规及合同范本、AI训练语料。每条知识均可自定义多标签。
3.7.2 知识入库管理
- 提交：用户填写元数据（标题、分类、标签、适用阶段、来源等），上传或编辑内容，案例需填写结构化字段。
- 脱敏：系统自动扫描敏感信息（单位名、人名、金额）并高亮，提交人确认脱敏或使用自动替换。
- 审核：知识管理员审核准确性、完整性、脱敏和分类，通过后正式发布。
- 主要操作：上传文件、在线编辑、从项目文档导入、脱敏检查、保存草稿、提交审核、AI提取案例要素等。
3.7.3 知识维护管理
- 修订：发起新版本，保留历史版本，审核后生效。
- 失效管理：管理员可标记失效并填写替代知识，系统通知相关用户。
- 评价反馈：用户可评分、评论，管理员定期分析优化。
- 主要操作：修订、标记失效、对比版本、恢复旧版、评分/评论。
3.7.4 智能问答应用
独立页面“监理智能助手”，基于RAG技术实现：
- 用户自然语言提问，系统从知识库检索相关条文，大模型生成答案并标明引用来源。
- 提供常用问题推荐、多轮对话、答案反馈（有帮助/无帮助）、问答记录导出。
- 超出知识域时明确提示，避免编造。
3.7.5 嵌入系统功能的AI智能应用
与各功能模块深度集成，均采用“生成建议→人工确认”模式：
- 合同智能审核（合同管理界面）
- 方案智能评审（方案审核界面）
- 监理意见辅助生成（通知单、联系单、会议纪要等编辑界面）
- 变更影响估算（变更分析界面）

3.8.4.2 异常事件处理流程
1. 事件登记/自动触发 → 系统生成事件编号，通知相关管理层（部门经理）和相关项目总监。
2. 事件评估与分派：部门经理在 24 小时内评估事件，指定处理责任人（可为总监、监理工程师或专项负责人），设定处理时限。若为“重大”等级事件，自动升级至分管副总知悉，并抄送公司负责人。
3. 制定处理方案：责任人在系统内编制《异常事件处理方案》，包括原因分析、处理措施、资源需求、预计完成时间，提交部门经理审批。
4. 处理执行：责任人按方案执行，在处理页面记录进展，上传阶段性证据。系统自动追踪时限，超时预警。
5. 处理结果验证：责任人提交《处理结果报告》，部门经理或其指定人验证结果是否有效。若未达到目标，退回重新处理或调整方案。
6. 事件关闭与复盘：验证通过后，部门经理确认关闭事件。系统提示可选择“沉淀为知识库案例”（脱敏后）。
7. 升级机制：若事件在规定处理时限内未关闭或处理效果不佳，系统自动升级至更高管理层（部门经理→分管副总），并以醒目通知提醒。
3.8.4.3 风险分析与管理
- 管理层可基于异常事件日志、项目健康度指标，开展风险识别与分析。
- 风险登记册：记录已识别风险（也可从异常事件中提取），包含风险描述、触发条件、影响分析、概率/影响评级、应对策略、责任人。
- 风险监控看板：以矩阵图展示高、中、低风险项数量与分布。
- 定期风险评估报告：系统可按月度/季度自动生成《项目风险分析报告》，汇总各部门/项目风险项、处置状态、趋势变化，供管理层会议使用。
3.8.4.4 相关表单与操作按钮
管理层事件/风险工作台操作按钮：

- 【待处理事件】：显示当前用户需要关注/审批的未关闭事件。
- 【事件升级处理】：仅分管副总可见，对超时未关闭或重复发生的事件进行强制干预。
- 【风险看板】：打开风险管理矩阵及统计图表。
- 【生成风险报告】：手动触发或按计划生成风险分析报告。

4. 非功能性需求
4.1 性能要求
- 并发用户数：支持 ≥100 在线用户。
- 页面响应时间：常规操作 ≤2秒，报表生成 ≤5秒。
- AI审核/问答接口：超时阈值 15秒，超时提示稍后重试。
4.2 可用性
- 系统可用性 ≥99.5%（按年度计）。
- 计划内维护需提前通知，维护窗口控制在凌晨时段。
- 故障恢复时间目标（RTO）<4小时，数据恢复点目标（RPO）<1小时。
4.3 安全性
- 传输加密：全站 HTTPS。
- 存储加密：敏感数据（如合同金额）数据库加密存储。
- 认证与访问控制：强密码策略，支持多因素认证（可选），严格 RBAC 权限模型。
- 审计日志：所有用户操作记录不可删除，保留至少 1 年。
- 安全合规：满足信息安全等级保护二级要求，上线前通过渗透测试。
4.4 可维护性与扩展性
- 采用模块化设计，预留移动端API接口。
- 支持配置化审批流程和阈值参数（如变更金额阈值、偏差报警比例）。
- 提供完善的系统监控与日志分析功能。
4.5 浏览器兼容性
- 支持 Chrome、Edge 最新版本，Firefox ESR。
4.6 数据备份与恢复
- 每日增量备份，每周全量备份，备份数据保留 3 个月。
- 定期进行恢复演练，确保数据可恢复。

5. 数据需求
5.1 核心数据实体
- 项目、合同、人员、文档、监理规划/细则、各类过程记录（通知单、日志、审核表等）、知识条目、AI反馈记录等。
5.2 数据关系
- 以项目为主线，挂接所有监理成果；合同关联支付与变更；人员关联项目与任务；知识独立维护，通过标签与引用关系与项目松散耦合。
5.3 数据迁移
- 若有旧系统，需提供数据模板与清洗规则，支持历史项目资料导入归档模块。

6. 接口需求
- 内部集成：与企业OA/HR系统对接组织架构与人员信息；与电子签章系统对接实现文件签章；与邮件/企业微信集成消息通知。
- 外部接口：预留与常见项目管理工具（如MS Project）的数据导入导出；归档包支持对接档案管理系统的标准格式。
- AI模型服务：采用内部私有化部署的大模型API，确保工程数据不出域。

7. 验收与交付
7.1 验收标准
- 功能测试用例通过率 100%，无阻塞性缺陷。
- 性能压测达到并发与响应时间要求。
- 安全扫描无高危漏洞，通过渗透测试。
- 用户UAT测试签字确认率 ≥95%。
7.2 交付物
- 系统安装部署包与源码。

8. 附录
8.1 审批阈值建议（待公司确认）
- 大型/重点项目规划需分管副总批准：合同额 >100万元或公司认定战略项目。
- 变更升级审批：费用增加 >10万元或工期延长 >15天。
- 进度偏差预警：关键路径延误 >5%总工期。
